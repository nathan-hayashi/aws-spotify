1. SSH KEY GENERATION & NAMING
================================================================================

WHAT HAPPENED:
  Ran: ssh-keygen -t ed25519 -C "aws-spotify-ec2" -f C:\dev\aws-spotify
  Result: Keys named "Cdevaws-spotify" and "Cdevaws-spotify.pub"

WHY:
  On Windows, ssh-keygen interpreted the backslash path C:\dev\aws-spotify as a
  relative filename, stripping the backslashes entirely. The tool concatenated
  the path components into one flat string: "Cdevaws-spotify".

HOW IT WAS FIXED:
  Renamed the files to follow standard conventions:
    ssh/id_ed25519       (private key)
    ssh/id_ed25519.pub   (public key)
  Stored in a dedicated ssh/ directory within the project.

LESSON LEARNED:
  Always use forward slashes (C:/dev/aws-spotify) or quoted paths when running
  commands in WSL/Git Bash that accept Windows paths. Better yet, use the full
  Linux path (/mnt/c/dev/aws-spotify/).

2. WSL PATH RESOLUTION (~/ vs /mnt/c/)
================================================================================

WHAT HAPPENED:
  Multiple commands failed with "No such file or directory":
    cd ~/aws-spotify
    cat > ~/aws-spotify/terraform/modules/ec2/main.tf << 'EOF'
    cat >> ~/aws-spotify/terraform/environments/dev/main.tf << 'EOF'

WHY:
  In WSL, ~ expands to /home/luckytuffy/ (the Linux home directory).
  The project lives on the Windows filesystem at /mnt/c/dev/aws-spotify/.
  These are two completely different mount points. ~/aws-spotify does not exist.

HOW IT WAS FIXED:
  Replaced all ~/aws-spotify references with /mnt/c/dev/aws-spotify/ in every
  command. Alternative fix: create a symlink:
    ln -s /mnt/c/dev/aws-spotify ~/aws-spotify

HOW MANY TIMES THIS HAPPENED:
  At least 8+ times across VPC module creation, EC2 module creation, and
  dev main.tf appends. This was the single most recurring issue in the project.

LESSON LEARNED:
  When working in WSL with files on the Windows filesystem, always use the full
  /mnt/c/ path. The ~ shortcut only works for files in the Linux home directory.
  Consider creating a symlink early in the project to avoid this entirely.

3. IAM POLICY ARN FORMATTING
================================================================================

WHAT HAPPENED:
  Commands failed with "NoSuchEntity" when trying to detach/delete IAM policies:
    arn:aws:iam::037336516853:policy/SpotifyDeveloperPolicy
  Error: Policy was not found.

WHY:
  The IAM ARN format requires double colons between "iam" and the account ID:
    arn:aws:iam::<ACCOUNT_ID>:policy/PolicyName
  The script had: arn:aws:iam:${ACCOUNT_ID}:policy/ (single colon before ${})
  instead of:     arn:aws:iam::${ACCOUNT_ID}:policy/ (double colon)

  Additionally, the $ACCOUNT_ID variable was sometimes empty when commands ran,
  shifting the account number into the wrong position.

HOW IT WAS FIXED:
  Used the correct ARN format with double colons:
    arn:aws:iam::${ACCOUNT_ID}:policy/SpotifyDeveloperPolicy

LESSON LEARNED:
  IAM ARNs always have an empty region field (between the two colons) because
  IAM is a global service. Always verify ARN format:
    arn:aws:iam::<account-id>:policy/<policy-name>
  Also verify $ACCOUNT_ID is populated before using it:
    echo $ACCOUNT_ID

4. S3 BUCKET CREATION — us-east-1 LOCATION CONSTRAINT
================================================================================

WHAT HAPPENED:
  Error: InvalidLocationConstraint when creating S3 bucket in us-east-1 with:
    --create-bucket-configuration LocationConstraint=us-east-1

WHY:
  us-east-1 is the default S3 region. AWS does not accept it as an explicit
  LocationConstraint — you must OMIT the --create-bucket-configuration flag
  entirely when creating buckets in us-east-1. Every other region requires it.

HOW IT WAS FIXED:
  Removed the --create-bucket-configuration flag:
    aws s3api create-bucket \
      --bucket spotify-cloudtrail-${ACCOUNT_ID} \
      --region us-east-1

LESSON LEARNED:
  us-east-1 is the one exception to S3 bucket creation. No LocationConstraint
  needed (or allowed). This is an AWS quirk worth memorizing.

5. IAM POLICY DELETION — DeleteConflict
================================================================================

WHAT HAPPENED:
  Error: Cannot delete a policy attached to entities.
  When trying to delete SpotifyPermissionBoundary.

WHY:
  A permission boundary is attached differently from a regular policy. It's set
  directly on IAM users/roles, not via group policy attachment. The script only
  detached group policies but didn't remove the permission boundary from roles.

HOW IT WAS FIXED:
  1. Listed entities the policy was attached to:
       aws iam list-entities-for-policy --policy-arn <arn>
  2. Removed the boundary from the role:
       aws iam delete-role-permissions-boundary --role-name SpotifyEC2Role
  3. Then deleted the policy:
       aws iam delete-policy --policy-arn <arn>

LESSON LEARNED:
  Permission boundaries are a separate attachment mechanism from regular
  policies. detach-group-policy won't remove them. Use:
    delete-user-permissions-boundary (for users)
    delete-role-permissions-boundary (for roles)

6. TERRAFORM MODULE DIRECTORY STRUCTURE
================================================================================

WHAT HAPPENED:
  Error: "No such file or directory" when writing to terraform/modules/vpc/main.tf

WHY:
  The directory terraform/modules/vpc/ didn't exist yet. cat > cannot create
  intermediate directories — only files. The parent directory must exist first.

HOW IT WAS FIXED:
  Created the directory first:
    mkdir -p terraform/modules/vpc

LESSON LEARNED:
  Always mkdir -p before writing files to new directories. The -p flag creates
  all intermediate directories and doesn't error if they already exist.
  Each Terraform module needs its own directory with at least one .tf file.

7. HEREDOC (cat << EOF) FAILURES
================================================================================

WHAT HAPPENED:
  Multiple heredoc commands produced mangled output or shell errors:
  - EC2 module: "EOFalue = aws_instance.api_server.private_ip"
  - Seed SQL: "SQLEOF song_id, artist_id FROM songs;ist_id)/1/ok.ogg', 28);"

WHY:
  The heredoc closing marker (EOF/SQLEOF) must appear on its OWN LINE with no
  other text. If the marker text appears embedded within the content, bash
  terminates the heredoc early, and everything after leaks out as raw shell
  commands. This causes:
  - Garbled output
  - Partially executed commands
  - Data corruption

HOW IT WAS FIXED:
  For the SQL seed data, switched to using a file instead of heredoc:
    nano /tmp/seed.sql   (paste SQL content)
    psql ... -f /tmp/seed.sql
  Removed stray SQLEOF markers from the file content.

LESSON LEARNED:
  For large multi-line content, write to a file with nano/vi instead of using
  heredocs. Heredocs are fragile with complex content (especially content
  containing the delimiter string). Using -f <file> with psql is more reliable
  than piping heredocs.

9. EC2 KEY PAIR NOT FOUND IN AWS
================================================================================

WHAT HAPPENED:
  Error: InvalidKeyPair.NotFound: The key pair 'aws-spotify-key' does not exist

WHY:
  Having SSH keys locally is not enough. The PUBLIC key must be imported into
  AWS EC2 so the service knows about it. The import command was initially
  pointing to the wrong path (~/.ssh/aws-spotify-key.pub).

HOW IT WAS FIXED:
  Imported the key with the correct path:
    aws ec2 import-key-pair \
      --key-name aws-spotify-key \
      --public-key-material fileb:///mnt/c/dev/aws-spotify/ssh/id_ed25519.pub

LESSON LEARNED:
  EC2 key pairs exist in two places:
  1. Your local machine (private + public key files)
  2. AWS EC2 (public key registered via import-key-pair or create-key-pair)
  Both must be set up. The --key-name is just a label in AWS — it doesn't need
  to match your local filename.

10. SSH KEY FILE PERMISSIONS (0777 ON WINDOWS FILESYSTEM)
================================================================================

WHAT HAPPENED:
  WARNING: UNPROTECTED PRIVATE KEY FILE!
  Permissions 0777 for '/mnt/c/dev/aws-spotify/ssh/id_ed25519' are too open.

WHY:
  Files on /mnt/c/ (Windows filesystem) default to 0777 permissions in WSL.
  SSH requires private keys to be readable only by the owner (0600 or 0400).
  chmod typically has no effect on Windows-mounted filesystems in WSL.

HOW IT WAS FIXED:
  Copied the key to the Linux filesystem where permissions work:
    cp /mnt/c/dev/aws-spotify/ssh/id_ed25519 ~/.ssh/id_ed25519_spotify
    chmod 600 ~/.ssh/id_ed25519_spotify

LESSON LEARNED:
  SSH keys used from WSL should live in ~/.ssh/ (Linux filesystem), not on
  /mnt/c/ (Windows filesystem), because Windows FS doesn't support Linux
  file permissions properly. Copy keys to ~/.ssh/ and chmod 600 them.

13. S3 LIFECYCLE RULE MISSING FILTER
================================================================================

WHAT HAPPENED:
  Warning: No attribute specified when one (and only one) of
  [rule[0].filter, rule[0].prefix] is required

WHY:
  AWS provider v5.x requires lifecycle rules to explicitly declare which objects
  they apply to. An empty filter {} means "apply to all objects." Without it,
  the provider warns now and will error in a future version.

HOW IT WAS FIXED:
  Added filter {} to the lifecycle rule block:
    rule {
      id     = "move-old-versions-to-ia"
      status = "Enabled"
      filter {}    <-- added this
      ...
    }

LESSON LEARNED:
  Always include filter {} in S3 lifecycle rules even if you want it to apply
  to all objects. It's explicit and future-proof.

14. sed APPLYING CHANGES TOO BROADLY
================================================================================

WHAT HAPPENED:
  Used sed to add "filter {}" after "rule {" but it matched EVERY "rule {" in
  the file — including encryption rules and CORS rules that don't support filter.
  Error: Unsupported block type "filter" in encryption and CORS resources.

WHY:
  sed 's/rule {/rule {\n    filter {}/' matched ALL occurrences of "rule {"
  in the file, not just the lifecycle rule. The encryption rule's "rule {" and
  the CORS "cors_rule {" (which contains "rule {") were also modified.

HOW IT WAS FIXED:
  Manually removed the incorrect filter {} from the encryption and CORS blocks,
  keeping only the one in the lifecycle rule.

LESSON LEARNED:
  sed is a blunt tool for editing structured files like Terraform HCL. It has
  no understanding of context — it just matches text patterns. For targeted
  edits in config files, use a proper editor (nano, vi, VS Code) or be
  extremely specific with sed patterns and line numbers.

15. CLOUDFRONT ORIGIN REQUIRES DOMAIN NAME, NOT IP
================================================================================

WHAT HAPPENED:
  Error: The parameter origin name cannot be an IP address.

WHY:
  CloudFront requires a DNS hostname for origins. The module was using
  var.ec2_public_ip (a raw IP like 34.195.227.70) as the domain_name for
  the EC2 API origin. CloudFront does not accept IP addresses.

HOW IT WAS FIXED:
  Changed three files to use the Elastic IP's public DNS name instead:
  1. EC2 module: added output "public_dns" { value = aws_eip.api.public_dns }
  2. CloudFront module: changed var.ec2_public_ip to var.ec2_public_dns
  3. Dev main.tf: passed module.ec2.public_dns instead of module.ec2.public_ip

LESSON LEARNED:
  CloudFront origins must be domain names, never IP addresses. Elastic IPs in
  AWS automatically get a public DNS name (ec2-X-X-X-X.compute-1.amazonaws.com)
  which can be used as the origin.

16. BASH HISTORY EXPANSION (! IN PASSWORDS)
================================================================================

WHAT HAPPENED:
  -bash: !': event not found
  When running: psql ... "CREATE USER spotify_app WITH PASSWORD 'SpotifyDev2026!';"

WHY:
  The ! character inside double quotes triggers bash history expansion. Bash
  sees !' and tries to interpret it as a reference to a previous command,
  which fails. Because the CREATE USER failed, subsequent GRANT commands
  also failed with "role does not exist."

HOW IT WAS FIXED:
  Disabled history expansion temporarily:
    set +H
    psql ... "CREATE USER spotify_app WITH PASSWORD 'SpotifyDev2026!';"
    set -H

LESSON LEARNED:
  Special characters in passwords (especially !) cause problems in bash when
  inside double quotes. Options:
  - Use set +H to disable history expansion
  - Use single quotes for the outer SQL
  - Escape the ! with a backslash
  - Use interactive psql \password command instead

17. POSTGRESQL citext EXTENSION NOT INSTALLED
================================================================================

WHAT HAPPENED:
  ERROR: extension "citext" is not available
  Could not open extension control file "citext.control": No such file or directory

WHY:
  The citext extension is part of the postgresql-contrib package, which was not
  included in the EC2 user_data install script. Only postgresql16-server and
  postgresql16 were installed.

WHAT IS citext:
  Case-insensitive text type for PostgreSQL. Treats "Nathan", "nathan", and
  "NATHAN" as the same value. Essential for email addresses and usernames to
  prevent duplicate accounts with different casing.

HOW IT WAS FIXED:
  Installed the contrib package:
    sudo dnf install -y postgresql16-contrib
  Then created the extension:
    psql spotify -c "CREATE EXTENSION IF NOT EXISTS citext;"

LESSON LEARNED:
  PostgreSQL extensions like citext, uuid-ossp, pgcrypto etc. require the
  -contrib package. Include it in the EC2 user_data script:
    dnf install -y postgresql16-server postgresql16 postgresql16-contrib

18. POSTGRESQL sudo PASSWORD (postgres USER)
================================================================================

WHAT HAPPENED:
  [sudo] password for postgres: sudo: a password is required
  When running sudo commands as the postgres user.

WHY:
  The postgres system user does not have passwordless sudo. Only ec2-user has
  that privilege on Amazon Linux by default. When switched to the postgres user
  via sudo -u postgres, you lose ec2-user's sudo access.

HOW IT WAS FIXED:
  Exited back to ec2-user first, then ran the command with sudo:
    exit   (back to ec2-user)
    sudo dnf install -y postgresql16-contrib

LESSON LEARNED:
  On Amazon Linux EC2, only ec2-user has passwordless sudo. If you need to
  install packages or modify system files, do it as ec2-user with sudo, not
  from within the postgres user session.

19. pg_hba.conf RULE ORDERING (FIRST MATCH WINS)
================================================================================

WHAT HAPPENED:
  FATAL: Ident authentication failed for user "spotify_app"
  Even after adding: host spotify spotify_app 127.0.0.1/32 md5

WHY:
  pg_hba.conf is processed top to bottom, FIRST MATCH WINS. The file had:
    host  all  all  127.0.0.1/32  ident     <-- matches first, uses ident
    host  spotify  spotify_app  127.0.0.1/32  md5   <-- never reached

  The broader "all all" rule matched before the specific "spotify spotify_app"
  rule, so PostgreSQL used ident authentication (which checks OS username)
  instead of md5 (password authentication). ec2-user != spotify_app, so it
  fails.

HOW IT WAS FIXED:
  Moved the specific rule ABOVE the generic rule in pg_hba.conf using vi:
    host  spotify  spotify_app  127.0.0.1/32  md5   <-- now matches first
    host  all      all          127.0.0.1/32  ident
  Then: sudo systemctl reload postgresql

LESSON LEARNED:
  pg_hba.conf works like a firewall — first matching rule wins. Always put
  specific rules ABOVE generic ones. When adding new rules, never just append
  to the end of the file — check what rules are above and whether they'll
  match first.

20. POSTGRESQL ROLE PERMISSIONS (TRUNCATE / SEQUENCES)
================================================================================

WHAT HAPPENED:
  ERROR: must be owner of sequence playlists_playlist_id_seq
  When running TRUNCATE ... RESTART IDENTITY CASCADE as spotify_app.

WHY:
  TRUNCATE with RESTART IDENTITY requires OWNERSHIP of the sequences (not just
  usage/select permissions). The tables and sequences are owned by postgres,
  and spotify_app only had CONNECT and USAGE grants.

  Additionally, running the seed file multiple times (because TRUNCATE kept
  failing) inserted duplicate data on top of existing data.

HOW IT WAS FIXED:
  Split the workflow:
  1. Truncate as postgres: sudo -u postgres psql spotify -c "TRUNCATE ..."
  2. Remove TRUNCATE from seed.sql
  3. Insert as spotify_app: psql ... -f /tmp/seed.sql
  Also granted broader permissions for future use:
    GRANT ALL ON ALL SEQUENCES IN SCHEMA public TO spotify_app;

LESSON LEARNED:
  TRUNCATE with RESTART IDENTITY requires table ownership, not just privileges.
  For application users that don't own the tables, either:
  - Truncate as the owner (postgres) separately
  - Use DELETE FROM instead of TRUNCATE (slower but doesn't need ownership)
  - Transfer ownership of tables to the app user

22. VS CODE TERRAFORM EXTENSION FALSE POSITIVES
================================================================================

WHAT HAPPENED:
  VS Code showed red errors:
    "Unexpected attribute: An attribute named 'project_name' is not expected here"
    "Unexpected attribute: An attribute named 'admin_ip' is not expected here"

WHY:
  The Terraform VS Code extension caches module schemas in the .terraform
  directory. When a new module is added (like the EC2 module), the extension
  doesn't know about the new module's variables until terraform init is re-run
  to download/index the module references.

HOW IT WAS FIXED:
  Ran terraform init to re-initialize modules, which updated the .terraform
  directory and resolved the VS Code errors.

LESSON LEARNED:
  After adding new module blocks to Terraform, always run terraform init before
  trusting VS Code's error highlighting. The extension needs the .terraform
  directory to be current.

SUMMARY OF RECURRING THEMES
================================================================================

1. PATH ISSUES (Most Frequent)
   ~/ vs /mnt/c/, backslashes vs forward slashes, wrong key paths.
   Root cause: WSL bridging two filesystems (Linux + Windows).
   Fix: Always use absolute /mnt/c/ paths for Windows files.

2. ORDER OF OPERATIONS
   pg_hba.conf rule ordering, granting permissions before using them,
   importing keys before referencing them in Terraform.
   Root cause: Not understanding that sequence matters in config files and
   cloud resource creation.

3. SCOPE CONFUSION (Local vs Remote)
   Running EC2 commands locally, running terraform output from wrong directory.
   Root cause: Multiple terminal sessions (local WSL vs SSH to EC2).
   Fix: Always check your prompt to know where you are.

4. STRING/SYNTAX SENSITIVITY
   AMI ID prefix, IAM ARN double colons, bash ! expansion, heredoc markers.
   Root cause: Cloud APIs and shell interpreters are extremely literal.
   Fix: Copy-paste IDs, use files instead of heredocs for complex content.

5. PERMISSIONS MODEL
   Permission boundaries vs policies, table ownership vs grants, Linux file
   permissions on Windows FS, ec2-user vs postgres sudo access.
   Root cause: Multiple layered permission systems each with different rules.

24. AWS CONFIG — NoAvailableConfigurationRecorderException & S3 DELIVERY POLICY
================================================================================
Date: 2026-03-24

WHAT HAPPENED:
  Four Config rules failed with:
    NoAvailableConfigurationRecorderException: You must create a configuration
    recorder before you can create or update a Config rule.
  Then when setting up the recorder and delivery channel:
    InsufficientDeliveryPolicyException: Insufficient delivery policy to s3
    bucket: spotify-cloudtrail-037336516853, unable to write to bucket.
  Then:
    NoAvailableDeliveryChannelException: Delivery channel is not available to
    start configuration recorder.

WHY:
  Two issues in sequence:
  1. AWS Config rules require a running Configuration Recorder. Without it,
     Config has nothing to evaluate rules against. The setup order is strict:
     recorder -> delivery channel -> start recorder -> create rules.
  2. The S3 bucket policy on spotify-cloudtrail-037336516853 only granted write
     permissions to CloudTrail (cloudtrail.amazonaws.com), not to AWS Config
     (config.amazonaws.com). Config is a separate service that needs its own
     S3 bucket policy statements (GetBucketAcl, ListBucket, PutObject).

HOW IT WAS FIXED:
  1. Updated the S3 bucket policy to include Config service permissions:
     - s3:GetBucketAcl for config.amazonaws.com
